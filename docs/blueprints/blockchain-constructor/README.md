# Blueprint: конструктор блокчейнов

Этот раздел хранит планировочные материалы для будущей нарезки работ в FCL по
направлению "конструктор блокчейнов".

Это **blueprint**, то есть проектная разметка будущего направления. Слово
оставлено как имя раздела в документации FCL. Это не текущий публичный
программный интерфейс FCL, не обещание конкретных имен библиотек и не перенос
Spring внутрь FCL.

## Источник

Исходный пакет документов скопирован из:

```text
vbytemaster/blockchain@223fd74
```

Локальная копия лежит в:

```text
docs/blueprints/blockchain-constructor/source/
```

Копия нужна, чтобы при планировании FCL не прыгать между репами. Источником
продуктовой архитектуры остается репа `blockchain`; этот раздел фиксирует,
какие части могут стать нейтральными строительными блоками FCL.

## Главная граница

FCL остается универсальным набором библиотек и прикладных слоев для сложного
программного обеспечения:

- сервисов;
- сетевых приложений;
- блокчейнов;
- DePIN-систем, то есть децентрализованных сетей физической инфраструктуры;
- других распределенных продуктов.

FCL не становится блокчейном. Spring-совместимость, перенос контрактов,
продуктовая сборка узла и Spring-специфичные форматы остаются вне FCL.

В FCL могут попадать только нейтральные строительные блоки:

- общие понятия предложения и голоса;
- правила кворума;
- правила окончательности;
- общий интерфейс механизма согласования;
- нейтральный вид состояния для согласования;
- результат исполнения для согласования;
- ядро Savanna без Spring-деталей;
- каркас сообщений согласования поверх уже существующей одноранговой сети FCL.

## Что здесь не фиксируется

Эти документы **не фиксируют** будущие имена C++ библиотек, целей сборки или
модулей.

Рабочие названия вроде "механизм согласования", "ядро Savanna", "вид состояния"
и "результат исполнения" используются только для планирования. Конкретные
имена `fcl_*` целей, C++ модулей и пространств имен должны утверждаться в
отдельных реализационных задачах.

## Правило переноса в FCL

Компонент можно рассматривать как кандидат в FCL только если:

1. Его можно описать без Spring-терминов.
2. Его можно протестировать без Spring-совместимой цепи.
3. Он полезен не только конкретному будущему блокчейну.
4. Он не заставляет неблокчейн-пользователей FCL видеть модель цепи.

Если хотя бы одно условие не выполняется, компонент остается в `blockchain` или
в продуктовой репе поверх него.
