# Порядок будущей нарезки

Документ задает порядок будущих работ по конструктору блокчейнов в FORGE.
Текущая ветка только размечает направление; код, CMake и публичные C++
программные интерфейсы не меняются.

## Нулевая фаза: зафиксировать blueprint

### Цель

Сделать документы доступными внутри FORGE и отделить будущую нарезку от области
текущего релиза.

### Результат

- Исходные документы из `blockchain@223fd74` скопированы в `source/`.
- FORGE-вводный документ объясняет, что это blueprint.
- Карта блоков работ лежит в `planning/forge-block-map.md`.
- Порядок работ лежит в этом документе.
- `docs/index.md` ссылается на blueprint.

### Готовность

- Изменены только документы.
- `docs/roadmap.md` не изменен.
- публичные интерфейсы ядра FORGE не изменены.

## Фаза 1: нейтральные понятия цепи

### Цель

Выделить минимальные типы-значения и статусы, которые нужны следующим фазам.

### Почему первая

Без общего словаря нельзя стабильно описать механизм согласования, вид
состояния и результат исполнения.

### Входы

- `source/docs/architecture/01-forge-boundary.md`;
- `source/docs/architecture/02-blockchain-constructor.md`;
- `planning/forge-block-map.md`.

### Выход

Отдельная реализационная задача с конкретным именем будущей библиотеки, C++
модулями и тестами.

### Стоп-условие

Если тип нельзя протестировать без Spring, он не идет в FORGE.

## Фаза 2: интерфейс механизма согласования

### Цель

Описать общий контракт между цепью и механизмом согласования.

### Почему после фазы 1

Интерфейс должен опираться на нейтральные понятия блока, голоса, статуса
проверки и окончательности.

### Выход

Реализационная задача на минимальный порт механизма согласования и тестовый
искусственный механизм.

### Стоп-условие

Если интерфейс требует Spring-таблицу или Spring-заголовок, граница выбрана
неверно.

## Фаза 3: вид состояния для согласования

### Цель

Дать механизму согласования доступ только для чтения к данным участников,
политики и окончательности.

### Почему после интерфейса

Сначала нужно понять, какие операции механизма согласования требуют состояния.

### Выход

Реализационная задача на порт состояния только для чтения и тестовую реализацию
без Spring.

### Стоп-условие

Если через порт можно менять состояние цепи, это не вид состояния, а скрытый
управляющий центр.

## Фаза 4: результат исполнения для согласования

### Цель

Описать данные, которые исполнитель передает механизму согласования после
проверки блока.

### Почему после вида состояния

Согласование получает данные из двух источников: состояние до/вокруг блока и
результат исполнения блока. Их нужно проектировать рядом, но не смешивать.

### Выход

Реализационная задача на нейтральный результат исполнения и статусы проверки.

### Стоп-условие

Если механизм согласования начинает запускать исполнение, граница сломана.

## Фаза 5: тестовый механизм согласования

### Цель

Проверить, что интерфейсы достаточно нейтральны до реализации Savanna.

### Почему перед Savanna

Тестовый механизм дешевле и позволяет поймать протекание Spring-деталей раньше,
чем появится сложный алгоритм.

### Выход

Реализационная задача на искусственный механизм согласования для модульных
тестов.

### Стоп-условие

Если тестовый механизм невозможно написать без Spring, предыдущие фазы нужно
пересмотреть.

## Фаза 6: общее ядро Savanna

### Цель

Реализовать Savanna как готовый повторно используемый механизм согласования.

### Почему после тестового механизма

К этому моменту должны быть проверены границы: голоса, кворум, окончательность,
вид состояния и результат исполнения.

### Выход

Реализационная задача на ядро Savanna без Spring-обвязки.

### Стоп-условие

Если ядро требует Spring-совместимые типы, реализацию нужно остановить и
вернуть границу в документацию.

## Фаза 7: сообщения согласования поверх одноранговой сети FORGE

### Цель

Описать и реализовать общий каркас сообщений согласования на существующем
сетевом слое FORGE.

### Почему после ядра

До ядра не ясно, какие сообщения действительно нужны алгоритму, а какие
относятся только к Spring-обвязке.

### Выход

Реализационная задача на нейтральные сообщения и проверки размера/формы.

### Стоп-условие

Если сообщение содержит Spring-специфичный формат, оно должно быть в
`blockchain`.

## Фаза 8: Spring-обвязка в `blockchain`

### Цель

Подключить Spring-совместимую цепь как первого внешнего потребителя новых
FORGE-блоков.

### Почему не в FORGE

Spring-обвязка переводит конкретные Spring-таблицы и форматы в нейтральный вид.
Это не общая обязанность FORGE.

### Выход

Отдельный план в репе `blockchain`.

### Стоп-условие

Если для Spring-обвязки нужно менять ядро Savanna Spring-терминами, ядро
недостаточно нейтрально.

## Фаза 9: пул транзакций и хранилище блоков

### Цель

Отдельно проверить, можно ли выделить нейтральные каркасы пула транзакций и
хранилища блоков.

### Почему последняя

Эти компоненты сильнее зависят от конкретной цепи. Их нельзя переносить в FORGE
раньше, чем появится опыт первого потребителя.

### Выход

Отдельные решения:

- оставить компонент в `blockchain`;
- вынести только порт;
- вынести каркас с внешними политиками;
- отложить перенос.

### Стоп-условие

Если компонент заставляет FORGE знать Spring-формат транзакции или журнала
блоков, он остается вне FORGE.

## Проверка каждой будущей фазы

Перед переносом любого блока в FORGE нужно пройти одинаковый фильтр:

1. Блок можно описать без Spring.
2. Блок можно протестировать без `blockchain`.
3. Блок полезен хотя бы двум классам потребителей FORGE.
4. Блок не меняет существующую одноранговую сеть FORGE без отдельной причины.
5. Блок не фиксирует имена будущих библиотек раньше реализации.

Если фильтр не пройден, блок остается blueprint-идеей или уходит в
`blockchain`.
