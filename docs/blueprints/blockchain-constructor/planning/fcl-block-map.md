# Карта будущих FCL-блоков

Документ размечает будущие блоки работ, которые могут дать FCL базу для
построения блокчейнов. Это не реализационный план и не публичный API. Имена
блоков рабочие.

## Общий принцип

FCL должен давать нейтральные строительные блоки. Репа `blockchain` должна
собирать из них конкретную Spring-совместимую цепь.

В FCL не переносим:

- Spring-таблицы;
- Spring-действия;
- Spring-совместимые заголовки блоков;
- перенос конкретных контрактов;
- продуктовую сборку узла;
- экономику или правила ресурсов конкретной сети.

## Блок 1: нейтральные понятия цепи

### Цель

Дать минимальный набор общих понятий, которые нужны механизму согласования и
планированию цепи, но не содержат Spring.

### Кандидаты

- идентификатор блока;
- номер блока;
- ссылка на родителя;
- статус наличия данных;
- статус проверки;
- статус окончательности;
- причина отказа;
- результат выбора основной ветки.

### Граница FCL

В FCL допустимы только типы-значения и статусы, которые можно применить к любой
цепи или реплицируемому журналу. Spring-совместимые структуры остаются в
`blockchain`.

### Проверка нейтральности

- Типы можно использовать в искусственной цепи без Spring.
- Тесты не требуют контрактов.
- В именах и полях нет Spring-, Storlane- и продуктовой терминологии.

## Блок 2: интерфейс механизма согласования

### Цель

Описать явную границу между цепью и механизмом согласования.

### Кандидаты операций

- проверить предложение блока;
- проверить голос;
- построить данные согласования для заголовка;
- вернуть обновление окончательности;
- вернуть решение выбора основной ветки;
- описать сообщения согласования.

### Граница FCL

FCL может владеть контрактом взаимодействия и общими статусами результата.
Конкретное чтение Spring-состояния, упаковка Spring-сообщений и внешний
программный интерфейс цепи остаются вне FCL.

### Проверка нейтральности

- Искусственный механизм можно подключить к тестовой цепи.
- Решения возвращаются явно, без скрытых побочных эффектов.
- Нет зависимости от Spring-форматов.

## Блок 3: вид состояния для согласования

### Цель

Дать механизму согласования узкий доступ только к тем данным состояния, которые
нужны для проверки голосов, кворума и окончательности.

### Кандидаты данных

- текущая политика участников;
- будущая политика участников;
- последний окончательный блок;
- текущая голова;
- параметры кворума;
- ключи участников;
- поколение политики.

### Граница FCL

FCL может определить нейтральный порт только для чтения. Перевод Spring-таблиц
в этот вид делает `blockchain`.

### Проверка нейтральности

- Порт только читает данные.
- Механизм согласования не может менять состояние через этот порт.
- Вид состояния можно построить из тестовых данных без Spring.

## Блок 4: результат исполнения для согласования

### Цель

Описать данные, которые исполнитель передает механизму согласования после
применения блока.

### Кандидаты данных

- корень транзакций;
- корень действий;
- сводка квитанций;
- признаки изменения политики участников;
- данные, которые должны попасть в заголовок;
- ошибка исполнения;
- статус проверки полезной нагрузки блока.

### Граница FCL

FCL может владеть общей формой результата и статусами проверки. Spring-формат
квитанций, порядок полей и контрактная машина остаются в `blockchain`.

### Проверка нейтральности

- Результат можно построить в тестовой цепи.
- Механизм согласования не запускает исполнение сам.
- Исполнитель не вызывает сетевое голосование.

## Блок 5: общее ядро Savanna

### Цель

Сделать Savanna готовым повторно используемым механизмом согласования.

### Кандидаты ответственности

- участники;
- веса;
- голоса;
- проверка подписей;
- кворум;
- предложения блоков;
- доказательство кворума;
- окончательность;
- выбор основной ветки;
- ошибки согласования.

### Граница FCL

В FCL может жить только ядро алгоритма. Spring-обвязка остается внешним
потребителем.

### Проверка нейтральности

- Ядро тестируется без Spring.
- Spring-обвязка не реализует кворум заново.
- Другой блокчейн может подать свои данные участников.

## Блок 6: сообщения согласования поверх одноранговой сети FCL

### Цель

Описать общий каркас сообщений механизма согласования поверх уже существующей
одноранговой сети FCL.

### Кандидаты сообщений

- предложение;
- голос;
- доказательство кворума;
- запрос недостающих данных;
- ответ с недостающими данными.

### Граница FCL

FCL может дать общий каркас сообщений и маршрутизации через существующий
сетевой слой. Spring-совместимая упаковка сообщений и конкретные темы цепи
остаются в `blockchain`.

### Проверка нейтральности

- Сообщения можно передать через текущую одноранговую сеть FCL.
- Базовая проверка размера и формы выполняется до дорогой проверки.
- Сообщения не содержат Spring-типов.

## Блок 7: Spring-обвязка как внешний потребитель

### Цель

Зафиксировать, что Spring-обвязка не является FCL-блоком, но должна быть учтена
как главный первый потребитель будущих FCL-примитивов.

### Ответственность `blockchain`

- чтение Spring-совместимой политики участников;
- перевод участников в нейтральный вид;
- построение данных Spring-совместимого заголовка;
- проверка Spring-совместимого блока;
- применение окончательности к цепи;
- упаковка сетевых сообщений цепи;
- совместимые ответы внешнего программного интерфейса.

### Проверка

- Spring-политика переводится в нейтральный вид.
- Блок с корректными данными проходит проверку через ядро Savanna.
- Блок с неверными данными отклоняется.
- Контрактная совместимость не ломается.

## Блок 8: пул транзакций как кандидат

### Цель

Проверить, можно ли выделить нейтральный каркас пула транзакций.

### Кандидаты ответственности

- прием;
- удаление дублей;
- срок жизни;
- приоритет;
- вытеснение;
- выдача кандидатов;
- реакция на принятие блока;
- реакция на окончательность.

### Риск

Пул транзакций быстро становится цепе-специфичным из-за правил ресурсов,
проверки подписей, ошибок и порядка выбора. Его нельзя переносить в FCL раньше
проверки нейтральности.

### Проверка нейтральности

- Каркас работает на искусственном типе транзакции.
- Spring-проверка транзакции подключается снаружи.
- Ошибки совместимости остаются в `blockchain`.

## Блок 9: хранилище блоков как кандидат

### Цель

Проверить, можно ли описать общий порт хранения блоков без правил выбора
основной ветки.

### Кандидаты ответственности

- запись блока;
- чтение по номеру;
- чтение по идентификатору;
- наличие блока;
- диапазон блоков;
- отметка необратимости;
- данные для повторного проигрывания.

### Риск

Хранилище легко смешать с выбором основной ветки. В FCL допустим только
нейтральный порт хранения; Spring-журнал и совместимость старых файлов остаются
в `blockchain`.

### Проверка нейтральности

- Можно хранить две ветки.
- Хранилище не выбирает основную ветку.
- Отметка необратимости не равна решению выбора ветки.
