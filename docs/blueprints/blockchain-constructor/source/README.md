# Конструктор блокчейнов на базе FCL

Эта репа фиксирует архитектуру будущего блокчейна, который должен заменить Spring
без переноса его монолитной внутренней структуры. Документы здесь нужны не для
общего описания идеи, а как инженерная основа: по ним дальше будут нарезаться
работы в FCL и в этой репе.

FCL здесь рассматривается как внешний фундамент: набор библиотек и прикладных
слоев для сложного программного обеспечения. FCL не должен становиться
блокчейном. Он должен оставаться полезным для сервисов, сетевых приложений,
блокчейнов, DePIN-систем и других распределенных продуктов.

## Что строим

Первая цель - Spring-совместимый блокчейн, собранный поверх FCL более чисто, чем
исходная реализация Spring.

Это означает:

- контракты должны быть переносимы;
- двоичный интерфейс контрактов должен быть совместим там, где это важно;
- формат действий, транзакций, блоков и подписываемых данных должен быть
  совместим там, где он виден контрактам, узлам и внешним инструментам;
- внутренний код не должен повторять монолитную структуру Spring;
- механизм согласования должен быть отдельным заменяемым компонентом;
- Savanna должна быть готовым повторно используемым механизмом, а не задачей,
  которую каждый пользователь конструктора пишет заново.

## Что означает "конструктор блокчейнов"

Конструктор блокчейнов - это не один готовый блокчейн и не набор
Spring-специфичных классов в FCL.

Под конструктором здесь понимается набор слоев:

1. FCL дает общий фундамент:
   - жизненный цикл приложения;
   - подключаемые модули;
   - программные интерфейсы;
   - одноранговую сеть;
   - асинхронное выполнение;
   - криптографию;
   - двоичную сериализацию;
   - конфигурацию и проверку настроек.
2. Будущие общие блоки FCL могут дать:
   - общий интерфейс механизма согласования;
   - общие примитивы голосования;
   - общие примитивы окончательности;
   - ядро Savanna без Spring-специфичных деталей.
3. Репа `blockchain` собирает конкретную цепь:
   - Spring-совместимую модель блоков и транзакций;
   - переносимость контрактов;
   - прикладные правила исполнения;
   - сетевые сообщения конкретной цепи;
   - совместимые программные интерфейсы;
   - Spring-совместимую обвязку над Savanna.

Главное правило:

```text
FCL дает строительные блоки.
blockchain собирает конкретную цепь.
Spring задает цель совместимости, но не диктует внутреннюю архитектуру.
```

## Почему блокчейн вынесен отдельно от FCL

Если положить Spring-совместимую цепь прямо в FCL, FCL быстро станет
блокчейн-проектом. Это сломает его главную ценность: быть нейтральным
фундаментом для разных классов сложного ПО.

Поэтому в FCL не должны жить:

- Spring-таблицы;
- Spring-специфичные заголовки блоков;
- имена контрактов конкретной цепи;
- продуктовые правила будущего блокчейна;
- совместимые программные интерфейсы конкретной сети;
- логика переноса конкретных контрактов.

В FCL могут жить только общие механизмы, которые пригодны не только нашему
блокчейну:

- приложение и его жизненный цикл;
- сеть;
- сообщения;
- протоколы;
- криптография;
- двоичная упаковка данных;
- общее состояние;
- общий механизм согласования;
- общая модель голосов и окончательности.
