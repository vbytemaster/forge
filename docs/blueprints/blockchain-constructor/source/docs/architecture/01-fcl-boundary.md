# Граница FCL

## Назначение документа

Документ фиксирует, где проходит граница между FCL и репой `blockchain`.

Это ключевое решение для всего проекта. Если границу провести слишком низко,
FCL превратится в блокчейн. Если провести слишком высоко, FCL не станет
конструктором, а будет только набором разрозненных сетевых библиотек.

Нужная середина такая:

```text
FCL - нейтральный фундамент и общие механизмы.
blockchain - конкретная сборка цепи и совместимость со Spring.
```

## Что такое FCL в этой архитектуре

FCL - это фундаментальные C++ библиотеки и прикладный слой для сложного
программного обеспечения.

FCL должен помогать строить:

- микросервисы;
- сетевые службы;
- одноранговые узлы;
- блокчейны;
- DePIN-системы, то есть децентрализованные сети физической инфраструктуры;
- операторские утилиты;
- серверные приложения;
- распределенные системы с несколькими участниками.

FCL не должен требовать, чтобы пользователь строил именно блокчейн. Пользователь
должен иметь возможность взять только криптографию, только одноранговую сеть,
только прикладный слой или только двоичную сериализацию.

## Что FCL уже дает

Текущая ветка FCL уже закрывает нижний слой конструктора:

- жизненный цикл приложения;
- подключаемые модули;
- программные интерфейсы;
- одноранговую сеть;
- публикацию и получение сетевых сообщений;
- обнаружение участников;
- асинхронное выполнение;
- криптографические примитивы;
- двоичную сериализацию;
- работу с настройками;
- проверку структурированных данных.

Для этого этапа одноранговую сеть считаем готовой основой. Ее не нужно
перепроектировать в документации конструктора блокчейнов. Будущие документы и
задачи должны потреблять этот слой, а не возвращаться к libp2p-донорству как к
первому шагу.

## Почему FCL не должен становиться блокчейном

Если FCL начнет содержать конкретную модель Spring, он потеряет нейтральность.
Тогда любой пользователь FCL, которому нужен обычный сетевой сервис, будет
видеть в публичной модели чужую цепь, контракты, таблицы и правила
окончательности.

Это плохо по нескольким причинам:

- FCL станет тяжелее для неблокчейн-проектов;
- публичные типы начнут отражать продуктовые решения одной цепи;
- общие библиотеки будут чаще ломаться из-за изменений конкретного блокчейна;
- тесты FCL начнут зависеть от Spring-совместимости;
- архитектура снова начнет склеиваться в монолит.

FCL должен быть похож на набор качественных строительных деталей. Блокчейн -
один из домов, который можно из них построить, но чертеж конкретного дома не
должен стать формой кирпича.

## Допустимые понятия в FCL

В FCL допустимы нейтральные понятия, которые применимы к разным классам
распределенных систем:

- приложение;
- служба;
- подключаемый модуль;
- порт взаимодействия;
- сообщение;
- протокол;
- узел;
- участник;
- соединение;
- поток сообщений;
- криптографический ключ;
- подпись;
- хеш;
- сериализация;
- состояние;
- снимок состояния;
- журнал изменений;
- предложение;
- голос;
- кворум;
- правило принятия решения;
- окончательность;
- проверка сообщения;
- ошибка протокола;
- метрика;
- диагностическое событие.

Такие понятия не привязывают FCL к Spring. Их можно использовать и в
блокчейне, и в распределенной базе, и в DePIN-сети, и в обычном кластере
сервисов.

## Недопустимые понятия в FCL

В FCL не должны попадать понятия, которые делают FCL частью конкретного
блокчейна:

- Spring-таблицы;
- Spring-действия;
- Spring-специфичные расширения заголовка блока;
- имена контрактов конкретной сети;
- правила ресурсов Spring;
- конкретный формат продукта поверх Spring;
- имена Storlane;
- прикладные роли будущего блокчейна;
- продуктовые программные интерфейсы;
- перенос конкретных контрактов;
- конкретная учетная запись развертывания;
- конкретная схема запуска сети.

Эти вещи должны жить в `blockchain` или в продуктовых репах поверх него.

## Три слоя ответственности

Архитектуру удобно читать через три слоя.

### 1. Общее ядро

Общее ядро - это FCL.

Оно отвечает за:

- выполнение приложения;
- сеть;
- сообщения;
- проверку данных;
- криптографию;
- двоичную упаковку;
- общие интерфейсы согласования;
- общие примитивы голосования;
- общие примитивы окончательности.

Общее ядро не знает, что такое Spring-контракт или Spring-таблица.

### 2. Прикладная сборка

Прикладная сборка - это `blockchain`.

Она отвечает за:

- модель цепи;
- правила транзакций;
- хранение блоков;
- пул транзакций;
- переносимость контрактов;
- Spring-совместимые структуры;
- Spring-совместимые программные интерфейсы;
- выбор конкретного механизма согласования;
- связывание всех компонентов в один узел.

Прикладная сборка может использовать FCL, но не должна требовать изменения FCL
для каждого продуктового правила.

### 3. Совместимость со Spring

Совместимость со Spring - это отдельная обвязка внутри `blockchain`.

Она отвечает за:

- совместимые форматы блоков и транзакций;
- совместимые подписи и хеши;
- совместимый двоичный интерфейс контрактов;
- совместимые таблицы и действия;
- совместимые ответы внешних интерфейсов;
- правила переноса контрактов.

Эта обвязка может быть большой, но она не должна протекать в FCL.

## Как FCL может быть полезен разным продуктам

Один и тот же нейтральный слой FCL должен обслуживать разные сценарии.

### Микросервис

Микросервису нужны:

- приложение;
- конфигурация;
- журналирование;
- программный интерфейс;
- фоновые задачи;
- проверка входных данных;
- безопасное завершение.

Ему не нужны блоки, цепь и контракты.

### DePIN-система

DePIN-системе нужны:

- одноранговая сеть;
- идентичность узлов;
- обмен сообщениями;
- криптографические доказательства;
- протоколы обнаружения;
- учет доступности или выполненной работы.

Она может использовать механизм согласования, но не обязана иметь Spring-модель.

### Блокчейн

Блокчейну нужны:

- состояние цепи;
- транзакции;
- блоки;
- механизм согласования;
- окончательность;
- выбор основной ветки;
- проверка подписи;
- сетевой обмен блоками и транзакциями.

Он использует больше слоев FCL, но все равно не должен превращать FCL в свою
внутреннюю модель.

## Практическое правило для будущей разработки

Перед переносом любого компонента в FCL нужно ответить на три вопроса:

1. Может ли этот компонент быть полезен не только Spring-совместимому
   блокчейну?
2. Можно ли описать его без Spring-, Storlane- и продуктовых терминов?
3. Можно ли протестировать его без запуска конкретной цепи?

Если ответ "нет" хотя бы на один вопрос, компонент должен остаться в
`blockchain`.

## Примеры границы

### Допустимо в FCL

- общий тип голоса;
- проверка кворума;
- правило "сообщение принято, если собрано достаточно подписей";
- очередь сетевых сообщений;
- интерфейс чтения состояния, нужного механизму согласования;
- общий движок Savanna без Spring-таблиц.

### Должно остаться в `blockchain`

- извлечение политики финализаторов из Spring-совместимого состояния;
- упаковка Spring-совместимого расширения заголовка;
- проверка Spring-совместимого действия контракта;
- маршрут внешнего программного интерфейса с Spring-совместимым ответом;
- перенос конкретного WASM-контракта.

## Критерий успеха

Граница проведена правильно, если:

- FCL можно использовать без блокчейна;
- `blockchain` можно собирать из FCL-компонентов;
- Spring-совместимость не загрязняет общие интерфейсы;
- Savanna может быть готовым компонентом конструктора;
- другой проект может взять Savanna из FCL без Spring;
- будущая цепь может заменить Savanna другим механизмом согласования без
  переписывания сети, программного интерфейса и исполнения транзакций.
