# Конструктор блокчейнов

## Назначение

Документ описывает, каким должен быть конструктор блокчейнов на базе FORGE.

Конструктор должен позволять собирать разные цепи из повторно используемых
частей. Первая практическая цель - собрать Spring-совместимую цепь, но сделать
это чище, чем исходный Spring.

Важно: это не порт Spring один к одному. Spring для нас - цель совместимости и
источник поведения, которое нужно сохранить. Он не является образцом внутренней
архитектуры.

## Основная идея

Блокчейн-узел нужно разложить на независимые компоненты:

- приложение;
- сеть;
- программные интерфейсы;
- криптография;
- двоичная сериализация;
- состояние;
- исполнение транзакций;
- пул транзакций;
- хранилище блоков;
- выбор основной ветки;
- производство блоков;
- механизм согласования;
- совместимость контрактов.

Каждый компонент должен иметь понятные входы, выходы и запреты. Тогда можно
заменить механизм согласования, не переписывая сеть, и изменить модель
приложения, не ломая общее ядро.

## Слои системы

```text
Продуктовая цепь
  Spring-совместимость
  Контракты и прикладные правила
  Сборка узла

Конструктор блокчейна
  Модель цепи
  Исполнение транзакций
  Пул транзакций
  Хранилище блоков
  Выбор основной ветки
  Производство блоков
  Обвязка механизма согласования

Будущие общие блоки FORGE
  Интерфейс механизма согласования
  Голоса
  Кворум
  Окончательность
  Общее ядро Savanna

Текущий FORGE
  Приложение
  Подключаемые модули
  Программные интерфейсы
  Одноранговая сеть
  Асинхронное выполнение
  Криптография
  Сериализация
  Конфигурация
```

Нижний слой уже существует в FORGE. Средний слой проектируется здесь. Верхний слой
будет конкретной Spring-совместимой цепью.

## Что дает FORGE

FORGE должен давать строительные блоки, а не готовую цепь.

### Приложение

FORGE отвечает за:

- запуск;
- остановку;
- порядок подключения модулей;
- обработку ошибок запуска;
- управляемое завершение;
- фоновые задачи;
- конфигурацию.

Узел блокчейна должен быть обычным FORGE-приложением. Механизм согласования,
сеть, программные интерфейсы и хранилища подключаются как отдельные части.

### Сеть

FORGE отвечает за одноранговую сеть:

- соединение участников;
- обмен сообщениями;
- публикацию сообщений;
- обнаружение участников;
- сетевые протоколы;
- контроль ресурсов.

Конструктор блокчейна не должен заново реализовывать базовую сеть. Он должен
определить, какие сообщения цепи идут через уже существующие сетевые механизмы.

### Программные интерфейсы

FORGE отвечает за типизированные программные интерфейсы:

- описание запроса;
- описание ответа;
- проверку входных данных;
- привязку к транспортному слою;
- единый жизненный цикл обработчиков.

Spring-совместимые внешние интерфейсы должны жить в `blockchain`, но опираться
на общий слой FORGE.

### Криптография

FORGE отвечает за общие криптографические операции:

- хеширование;
- подписи;
- проверку подписей;
- ключи;
- BLS-подписи, если они используются механизмом согласования.

Цепь отвечает за то, какие именно байты подписываются и какие правила проверки
считаются совместимыми со Spring.

### Сериализация

FORGE отвечает за двоичную упаковку и распаковку данных.

Для Spring-совместимости критично, чтобы видимые форматы совпадали побайтно:

- действие контракта;
- транзакция;
- подпись транзакции;
- заголовок блока;
- подписываемая часть блока;
- данные, участвующие в хеше.

Общая упаковка может жить в FORGE. Spring-совместимый порядок полей и конкретные
структуры должны жить в `blockchain`.

## Что дает `blockchain`

Репа `blockchain` должна дать конкретную сборку цепи.

### Модель цепи

Модель цепи описывает:

- идентификатор цепи;
- номер блока;
- идентификатор блока;
- заголовок блока;
- подписанный блок;
- транзакцию;
- квитанцию исполнения;
- состояние блока;
- состояние окончательности.

Модель цепи должна отделять общие понятия от Spring-совместимых форматов.

### Исполнение транзакций

Исполнение транзакций отвечает за:

- проверку транзакции перед выполнением;
- применение действий;
- запуск контрактного кода;
- изменение состояния;
- создание квитанций;
- расчет корней данных;
- откат при ошибке.

Исполнение не должно решать, какая ветка цепи основная. Это зона выбора ветки и
механизма согласования.

### Пул транзакций

Пул транзакций отвечает за:

- прием транзакций до включения в блок;
- предварительную проверку;
- удаление дублей;
- порядок выбора;
- вытеснение старых или невалидных транзакций;
- передачу транзакций в сеть;
- передачу кандидатов производителю блока.

Пул транзакций не должен исполнять блоки и не должен принимать решения об
окончательности.

### Хранилище блоков

Хранилище блоков отвечает за:

- запись блоков;
- чтение блоков по номеру;
- чтение блоков по идентификатору;
- хранение обратимых блоков;
- хранение необратимых блоков;
- снимки состояния;
- данные для повторного проигрывания.

Хранилище блоков не должно выбирать основную ветку. Оно хранит данные, а не
принимает консенсусные решения.

### Выбор основной ветки

Выбор основной ветки отвечает за то, какая ветка считается принятой узлом.

Для разных механизмов согласования это правило может быть разным:

- у одной цепи решающим может быть накопленный вес;
- у другой - кворум голосов;
- у третьей - подтвержденная окончательность.

Поэтому выбор основной ветки должен быть связан с механизмом согласования, но
не должен быть спрятан в хранилище блоков.

### Производство блоков

Производство блоков отвечает за:

- выбор родительского блока;
- выбор транзакций из пула;
- запуск исполнения;
- сбор заголовка;
- получение данных механизма согласования;
- подпись блока;
- публикацию блока в сеть.

Производитель не должен сам реализовывать правила окончательности. Он должен
обращаться к механизму согласования через явный интерфейс.

### Механизм согласования

Механизм согласования отвечает за:

- проверку предложений блоков;
- проверку голосов;
- расчет кворума;
- обновление состояния окончательности;
- выбор допустимого родителя;
- правила основной ветки;
- сетевые сообщения, относящиеся к голосованию.

Он не должен напрямую знать внутренности Spring-совместимых таблиц. Для этого
нужна обвязка.

## Как менять механизм согласования

Механизм согласования должен подключаться через явную границу.

Минимальная логика замены:

1. Узел получает или создает блок.
2. Исполнение считает результаты блока.
3. Механизм согласования получает только те данные, которые нужны для проверки.
4. Механизм согласования возвращает решение:
   - блок допустим;
   - блок отклонен;
   - блок требует ожидания дополнительных данных;
   - блок меняет состояние окончательности;
   - основная ветка должна измениться.
5. Узел применяет решение через общий слой цепи.

Если эта граница выдержана, можно заменить Savanna другим механизмом без
переписывания пула транзакций, сетевого транспорта и внешних интерфейсов.

## Как собирать разные блокчейны

Разные цепи должны отличаться сборкой, а не копированием всего узла.

Пример различий:

- другая модель контрактов;
- другой механизм согласования;
- другой формат транзакций;
- другие правила ресурсов;
- другой внешний программный интерфейс;
- другая политика производителей блоков.

Пример общих частей:

- запуск приложения;
- сеть;
- криптография;
- двоичная упаковка;
- механизм голосования;
- журнал блоков;
- пул транзакций как каркас;
- наблюдаемость;
- конфигурация.

## Почему это не порт Spring один к одному

Spring в текущем виде смешивает слишком много ролей:

- управление цепью;
- исполнение транзакций;
- хранение блоков;
- выбор ветки;
- производство блоков;
- сетевой обмен;
- внешние интерфейсы;
- механизм окончательности;
- совместимость контрактов.

Если перенести это один к одному, получится новый монолит с другим названием.
Мы должны сохранить совместимость там, где она важна внешне, но заменить
внутреннее устройство.

Правильная цель:

```text
Снаружи - совместимость там, где она нужна.
Внутри - чистая разборная архитектура.
```

## Обязательные свойства конструктора

Конструктор считается правильным, если он:

- не требует Spring-терминов в FORGE;
- позволяет использовать готовую Savanna;
- позволяет заменить Savanna другим механизмом;
- отделяет хранение блоков от выбора ветки;
- отделяет пул транзакций от исполнения;
- отделяет исполнение от сетевого транспорта;
- делает связь согласования и состояния явной;
- дает понятные точки тестирования;
- сохраняет переносимость контрактов;
- не превращает совместимость в архитектурную грязь.

## Минимальный первый результат

Первый инженерный результат после документации должен быть не полный узел, а
разрез по компонентам:

- список будущих общих интерфейсов;
- список Spring-совместимых адаптеров;
- список потоков данных;
- тестовые требования для каждого слоя;
- порядок переноса в FORGE;
- критерии, по которым компонент считается достаточно общим для FORGE.
