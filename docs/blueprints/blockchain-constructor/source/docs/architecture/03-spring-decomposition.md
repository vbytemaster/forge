# Разбор Spring на компоненты

## Назначение

Spring рассматривается как цель совместимости, но не как архитектурный образец.
В нем исторически смешаны роли, которые в новом конструкторе должны стать
отдельными компонентами.

Цель документа - разложить Spring-монолит на части, которые можно проектировать,
тестировать и переносить отдельно.

## Главная проблема монолита

В монолитной реализации один центральный слой часто знает слишком много:

- как устроен блок;
- как исполняется транзакция;
- как хранится состояние;
- как выбирается ветка;
- как принимаются голоса;
- как публикуются блоки;
- как отвечают внешние интерфейсы;
- как контракт видит данные.

Такой узел сложно менять. Замена механизма согласования тянет за собой
исполнение, сеть, хранение блоков и внешние интерфейсы.

Новый дизайн должен сделать зависимости явными.

## Карта компонентов

```text
Модель цепи
  -> Хранилище блоков
  -> Исполнение транзакций
  -> Пул транзакций
  -> Производство блоков
  -> Выбор основной ветки
  -> Механизм согласования
  -> Сетевой обмен
  -> Прикладной программный интерфейс
  -> Совместимость контрактов
```

Ни один компонент не должен становиться новым скрытым управляющим монолитом.

## Модель цепи

### Назначение

Модель цепи задает основные типы, которыми оперируют остальные компоненты.

### Обязанности

- описывать идентификатор цепи;
- описывать номер блока;
- описывать идентификатор блока;
- описывать заголовок блока;
- описывать подписанный блок;
- описывать транзакцию;
- описывать результат исполнения;
- описывать статус блока;
- описывать признаки окончательности.

### Запреты

Модель цепи не должна:

- ходить в сеть;
- читать базу данных;
- исполнять контракт;
- выбирать основную ветку;
- проверять кворум;
- содержать продуктовую бизнес-логику.

### Входы

- байты из сети;
- байты из хранилища;
- данные от производителя блока;
- данные от исполнения.

### Выходы

- типизированные структуры;
- байты для подписи;
- байты для хеша;
- данные для внешнего интерфейса.

### Зависимости

- двоичная сериализация;
- криптографические хеши;
- правила Spring-совместимого порядка полей.

### Где живет

Общая форма может быть описана как часть конструктора, но Spring-совместимые
типы живут в `blockchain`.

## Хранилище блоков

### Назначение

Хранилище блоков сохраняет и отдает блоки и связанные служебные данные.

### Обязанности

- записывать блоки;
- читать блоки по номеру;
- читать блоки по идентификатору;
- хранить необратимые блоки;
- хранить обратимые блоки;
- хранить служебные отметки для повторного проигрывания;
- обеспечивать целостность записанных данных;
- давать снимки для восстановления.

### Запреты

Хранилище блоков не должно:

- решать, какая ветка основная;
- проверять голоса;
- исполнять транзакции;
- принимать сетевые сообщения;
- менять правила окончательности.

### Входы

- блоки от сети;
- блоки от производителя;
- отметки об окончательности;
- команды восстановления.

### Выходы

- блок по номеру;
- блок по идентификатору;
- диапазон блоков;
- служебные данные для восстановления;
- информация о наличии блока.

### Зависимости

- локальное долговременное хранилище;
- двоичная сериализация;
- проверка целостности.

### Где живет

Общий порт хранилища может стать частью конструктора. Spring-совместимый журнал
блоков и конкретный формат записи живут в `blockchain`.

## Исполнение транзакций

### Назначение

Исполнение транзакций применяет транзакции к состоянию цепи.

### Обязанности

- проверять структуру транзакции;
- проверять подписи и полномочия;
- выполнять действия;
- запускать контрактный код;
- менять состояние;
- считать квитанции;
- считать корни данных;
- фиксировать успешный блок;
- откатывать неуспешный блок.

### Запреты

Исполнение не должно:

- выбирать основную ветку;
- публиковать блоки в сеть;
- принимать голоса;
- хранить сетевые соединения;
- решать, кто следующий производитель, если это правило принадлежит механизму
  согласования.

### Входы

- транзакции;
- состояние перед блоком;
- ограничения исполнения;
- данные заголовка блока;
- сведения от механизма согласования, если они нужны контрактам.

### Выходы

- новое состояние;
- квитанции;
- корень транзакций;
- корень действий;
- данные для заголовка;
- ошибки исполнения.

### Зависимости

- состояние цепи;
- контрактная машина;
- правила авторизации;
- правила ресурсов;
- Spring-совместимый двоичный интерфейс контрактов.

### Где живет

Каркас исполнения может быть описан как компонент конструктора. Spring-правила
контрактов, ресурсов и авторизации живут в `blockchain`.

## Пул транзакций

### Назначение

Пул транзакций хранит транзакции, которые еще не включены в принятый блок.

### Обязанности

- принимать транзакции от внешнего интерфейса;
- принимать транзакции из сети;
- отбрасывать дубли;
- выполнять легкую предварительную проверку;
- учитывать срок жизни транзакций;
- выбирать кандидатов для блока;
- удалять транзакции после окончательности;
- передавать транзакции в сетевой слой.

### Запреты

Пул транзакций не должен:

- исполнять полный блок;
- менять долговременное состояние цепи;
- принимать решение об окончательности;
- выбирать основную ветку;
- знать внутренние детали сетевого транспорта.

### Входы

- транзакции от пользователя;
- транзакции от других узлов;
- сведения о принятом блоке;
- сведения об отклоненном блоке;
- сведения об окончательности.

### Выходы

- кандидатный набор транзакций;
- транзакции для сетевой публикации;
- отказ в приеме транзакции;
- статистика пула.

### Зависимости

- легкая проверка транзакции;
- часы;
- настройки лимитов;
- сведения о текущей основной ветке.

### Где живет

Общий каркас может стать частью конструктора. Spring-специфичная проверка
транзакций живет в `blockchain`.

## Производство блоков

### Назначение

Производитель блока собирает новый блок из текущего состояния и транзакций.

### Обязанности

- определить, можно ли производить блок сейчас;
- выбрать родителя;
- получить кандидатные транзакции из пула;
- запустить исполнение;
- получить данные механизма согласования;
- собрать заголовок;
- подписать блок;
- передать блок в сетевой слой;
- передать блок в локальную обработку.

### Запреты

Производитель не должен:

- сам проверять кворум;
- сам хранить историю голосов;
- напрямую менять правила окончательности;
- обходить исполнитель транзакций;
- писать в хранилище в обход общего пути приема блока.

### Входы

- текущее состояние цепи;
- текущая основная ветка;
- транзакции из пула;
- расписание или политика производства;
- данные механизма согласования.

### Выходы

- предложенный блок;
- ошибка невозможности производства;
- диагностические сведения.

### Зависимости

- пул транзакций;
- исполнение;
- механизм согласования;
- криптография;
- хранилище блоков.

### Где живет

Общий каркас производства может быть частью конструктора. Spring-совместимые
правила производителя живут в `blockchain`.

## Выбор основной ветки

### Назначение

Выбор основной ветки определяет, какую ветку локальный узел считает принятой.

### Обязанности

- сравнивать допустимые ветки;
- учитывать окончательность;
- учитывать решение механизма согласования;
- запрещать откат за окончательный блок;
- сообщать другим компонентам об изменении головы цепи.

### Запреты

Выбор ветки не должен:

- хранить блоки как долговременное хранилище;
- исполнять транзакции;
- принимать транзакции от пользователя;
- публиковать сетевые сообщения;
- содержать Spring-специфичные таблицы без адаптера.

### Входы

- блоки из хранилища;
- результаты проверки блока;
- состояние окончательности;
- голоса и решения механизма согласования.

### Выходы

- новая голова цепи;
- отказ переключения;
- требование догрузить блоки;
- событие развилки.

### Зависимости

- хранилище блоков;
- механизм согласования;
- состояние окончательности.

### Где живет

Общий интерфейс выбора ветки должен быть рядом с механизмом согласования.
Spring-совместимые правила - в `blockchain`.

## Механизм согласования

### Назначение

Механизм согласования отвечает за то, как участники сети приходят к общему
решению о блоках и окончательности.

### Обязанности

- проверять предложения блоков;
- проверять голоса;
- собирать кворум;
- определять окончательные блоки;
- участвовать в выборе основной ветки;
- давать данные для заголовка блока;
- определять сетевые сообщения голосования.

### Запреты

Механизм согласования не должен:

- напрямую исполнять контракты;
- хранить Spring-таблицы как свои внутренние типы;
- обходить состояние цепи;
- владеть внешним программным интерфейсом узла;
- подменять пул транзакций.

### Входы

- предложение блока;
- голоса;
- сведения о состоянии цепи;
- результат исполнения блока;
- сведения о политиках участников.

### Выходы

- решение о допустимости блока;
- обновление окончательности;
- данные для заголовка;
- сообщения голосования;
- решение выбора ветки.

### Зависимости

- криптография;
- состояние участников;
- сетевые сообщения;
- данные исполнения.

### Где живет

Общее ядро Savanna в будущем может жить в FCL. Spring-обвязка живет в
`blockchain`.

## Сетевой обмен

### Назначение

Сетевой обмен переносит блоки, транзакции, голоса и служебные сообщения между
узлами.

### Обязанности

- принимать сообщения;
- проверять базовый формат сообщений;
- ограничивать ресурсы;
- передавать блоки в обработчик блоков;
- передавать транзакции в пул;
- передавать голоса в механизм согласования;
- публиковать локальные блоки и голоса.

### Запреты

Сеть не должна:

- исполнять транзакции;
- принимать решение об окончательности;
- менять состояние цепи напрямую;
- знать внутреннюю схему контрактов;
- подменять прикладную проверку сообщения.

### Входы

- сообщения от других узлов;
- локальные сообщения для публикации;
- настройки сетевых тем и протоколов.

### Выходы

- типизированные входящие события;
- опубликованные сообщения;
- отказ по сетевой политике.

### Зависимости

- текущий слой одноранговой сети FCL;
- кодирование сообщений;
- политика ограничений.

### Где живет

Базовая сеть живет в FCL. Сообщения конкретной цепи и их маршрутизация живут в
`blockchain`.

## Прикладной программный интерфейс

### Назначение

Прикладной программный интерфейс дает внешним клиентам доступ к узлу.

### Обязанности

- принимать транзакции;
- отдавать блоки;
- отдавать состояние;
- отдавать сведения о цепи;
- отдавать сведения о производстве блоков;
- поддерживать совместимые ответы там, где это требуется для Spring.

### Запреты

Прикладной программный интерфейс не должен:

- исполнять транзакции в обход общего слоя;
- читать состояние в небезопасной форме;
- принимать решения механизма согласования;
- менять формат совместимости без явного решения.

### Входы

- запросы клиентов;
- настройки совместимости;
- доступ к состоянию;
- доступ к хранилищу блоков.

### Выходы

- ответы клиентам;
- транзакции в пул;
- ошибки проверки запроса.

### Зависимости

- слой программных интерфейсов FCL;
- состояние;
- пул транзакций;
- хранилище блоков;
- Spring-совместимые форматы ответов.

### Где живет

Общий слой программных интерфейсов живет в FCL. Конкретные маршруты и
совместимые ответы живут в `blockchain`.

## Совместимость контрактов

### Назначение

Совместимость контрактов обеспечивает перенос Spring-контрактов в новую цепь.

### Обязанности

- сохранить совместимый двоичный интерфейс;
- сохранить порядок полей в действиях;
- сохранить правила упаковки данных;
- сохранить видимые контрактам системные данные;
- сохранить правила авторизации там, где это критично;
- дать тесты переносимости.

### Запреты

Совместимость контрактов не должна:

- определять общую архитектуру FCL;
- заставлять все цепи использовать Spring-модель;
- протекать в общее ядро Savanna;
- смешиваться с сетевым транспортом.

### Входы

- контрактный код;
- действие;
- таблицы состояния;
- данные транзакции;
- данные блока.

### Выходы

- результат выполнения действия;
- изменение состояния;
- квитанции;
- ошибки совместимости.

### Зависимости

- двоичная сериализация;
- контрактная машина;
- Spring-совместимые типы;
- тесты старого поведения.

### Где живет

Только в `blockchain` и будущих продуктовых репах. В FCL могут жить лишь общие
инструменты упаковки, криптографии и исполнения приложения.

## Итоговый принцип

Каждый компонент должен иметь одну главную работу. Если компонент начинает
выполнять две роли, его нужно разделить раньше, чем он станет новым Spring.
