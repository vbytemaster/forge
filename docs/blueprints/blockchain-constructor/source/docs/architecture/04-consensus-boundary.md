# Граница согласования

## Назначение

Документ описывает границу между механизмом согласования, исполнением
транзакций и состоянием цепи.

Эта граница важна, потому что именно здесь Spring-монолит сложнее всего
разобрать. Нельзя просто сказать, что механизм согласования полностью отдельно
от состояния. Для Savanna и Spring-совместимой цепи это будет неправдой.

Нужно сделать связность явной, а не спрятанной.

## Что такое механизм согласования

Механизм согласования - это часть узла, которая помогает участникам сети прийти
к одному решению о блоках.

Он отвечает за:

- допустимость предложения блока;
- голоса участников;
- проверку подписей;
- кворум;
- обновление окончательности;
- выбор допустимого родителя;
- защиту от принятия конфликтующих веток;
- правила перехода основной ветки.

Механизм согласования не является всем блокчейном. Он не должен становиться
исполнителем контрактов, внешним программным интерфейсом и хранилищем блоков
одновременно.

## Почему граница не абсолютно чистая

В идеальной схеме механизм согласования получает блок, проверяет подписи и
возвращает "принять" или "отклонить". Для реального Spring-совместимого узла
этого мало.

Механизм согласования зависит от состояния цепи:

- кто имеет право голосовать;
- какая политика участников активна;
- какая политика вступит в силу позже;
- какие блоки уже окончательны;
- какие данные заголовка обязательны;
- какие предыдущие решения уже приняты.

Исполнение тоже зависит от согласования:

- часть данных блока может быть проверена механизмом согласования;
- результат исполнения входит в данные, которые подписываются;
- контракт может видеть данные блока;
- изменение состояния может менять будущий набор участников.

Поэтому правильная цель - не "полное разделение", а явные порты между слоями.

## Три стороны границы

```text
Состояние цепи
  -> читает исполнитель
  -> читает обвязка согласования
  -> хранит политики участников

Исполнение
  -> применяет транзакции
  -> получает результат блока
  -> формирует корни данных

Механизм согласования
  -> проверяет предложения и голоса
  -> обновляет окончательность
  -> принимает решения по веткам
```

Ни одна сторона не должна напрямую владеть всеми остальными.

## Данные, которые согласование читает из состояния

Механизм согласования не должен читать произвольное состояние. Ему нужен узкий
вид состояния.

Такой вид должен отдавать:

- текущую политику участников;
- будущую политику участников, если она уже запланирована;
- номер поколения политики;
- последний окончательный блок;
- текущую голову основной ветки;
- сведения о допустимых производителях;
- параметры кворума;
- сведения о ранее принятых голосах, если они являются частью состояния;
- данные, нужные для проверки подписи участника.

Этот вид состояния должен быть только для чтения. Механизм согласования не
должен менять состояние напрямую.

## Данные, которые исполнение передает в согласование

После исполнения блока механизм согласования должен получить результат, который
влияет на проверку блока.

Минимально нужны:

- корень транзакций;
- корень действий;
- сводка квитанций;
- изменения протокольных признаков;
- данные, которые должны попасть в заголовок;
- сведения о том, меняется ли политика участников;
- ошибки исполнения.

Механизм согласования не должен запускать исполнение сам. Он получает результат
через явную структуру.

## Данные, которые согласование возвращает узлу

Механизм согласования должен возвращать не "магические" побочные эффекты, а
явные решения:

- предложение блока принято;
- предложение блока отклонено;
- не хватает данных для проверки;
- голос принят;
- голос отклонен;
- кворум собран;
- блок стал окончательным;
- основная ветка должна переключиться;
- сообщение нужно опубликовать в сеть.

Такой результат можно тестировать без запуска полного узла.

## Предлагаемые порты

Названия ниже являются архитектурными, не окончательными именами кода.

### Вид состояния для согласования

Назначение: дать механизму согласования только те данные состояния, которые ему
нужны.

Обязанности:

- читать политику участников;
- читать последний окончательный блок;
- читать текущую голову;
- читать параметры кворума;
- читать ключи участников;
- читать отложенные изменения политики.

Запреты:

- менять состояние;
- исполнять транзакции;
- писать блоки;
- обращаться к сети.

### Вид результата исполнения

Назначение: передать механизму согласования результат уже выполненного блока.

Обязанности:

- предоставить корни данных;
- предоставить сводку квитанций;
- предоставить признаки изменения политики;
- предоставить ошибки исполнения;
- предоставить данные для заголовка.

Запреты:

- хранить полный исполнитель;
- давать доступ к произвольной базе;
- позволять механизму согласования перезапустить исполнение.

### Движок согласования

Назначение: принимать решения по предложениям, голосам, окончательности и веткам.

Обязанности:

- проверить предложение блока;
- проверить голос;
- собрать кворум;
- построить данные согласования для заголовка;
- вернуть обновление окончательности;
- вернуть решение выбора ветки;
- указать сетевые сообщения.

Запреты:

- знать Spring-таблицы напрямую;
- выполнять контракты;
- писать в хранилище блоков;
- принимать запросы внешнего программного интерфейса.

## Обработка предложения блока

Поток должен выглядеть так:

```text
Получен блок
  -> Проверена базовая форма
  -> Проверен родитель
  -> Исполнение применило транзакции во временном состоянии
  -> Получен результат исполнения
  -> Механизм согласования проверил заголовок и данные голосования
  -> Выбор ветки принял решение
  -> Состояние зафиксировано или откатано
```

Важный запрет: механизм согласования не должен тайно исполнять транзакции внутри
себя.

## Обработка голоса

Поток должен выглядеть так:

```text
Получен голос
  -> Сеть проверила базовый формат
  -> Обвязка цепи определила, к какому блоку относится голос
  -> Вид состояния дал данные участника
  -> Механизм согласования проверил подпись и право голоса
  -> Голос принят или отклонен
  -> При кворуме обновлена окончательность
  -> Узел опубликовал нужные последствия
```

Голос не должен напрямую менять состояние в обход решения механизма
согласования.

## Обработка изменения политики участников

Для Savanna важны политики участников. Они могут зависеть от состояния цепи.

Поток должен быть явным:

```text
Транзакция изменила состояние
  -> Исполнение выявило изменение политики
  -> Изменение зафиксировано как будущая политика
  -> Механизм согласования видит ее через вид состояния
  -> На нужной высоте или поколении политика становится активной
```

Запрещено, чтобы механизм согласования сам искал Spring-таблицу и сам
интерпретировал ее как внутренний формат.

## Как не вернуть монолитный управляющий центр

Есть несколько опасных признаков.

### Признак 1: один класс знает все

Если один объект содержит:

- хранилище блоков;
- состояние;
- сеть;
- пул транзакций;
- голоса;
- производство блоков;
- внешний программный интерфейс;

это новый монолит.

### Признак 2: хранилище выбирает ветку

Хранилище может знать, какие блоки есть. Оно не должно владеть правилами
основной ветки.

### Признак 3: сеть исполняет сообщения

Сеть может доставить сообщение и выполнить базовую защиту от мусора. Она не
должна применять блок к состоянию.

### Признак 4: исполнитель принимает голоса

Исполнитель применяет транзакции. Голоса принадлежат механизму согласования.

### Признак 5: Spring-совместимость попала в ядро Savanna

Если общее ядро Savanna знает конкретные Spring-таблицы, оно уже не общее.

## Проверка границы

Граница считается правильной, если можно написать отдельные тесты:

- механизм согласования проверяет голос на искусственном состоянии;
- исполнитель применяет блок без реальной сети;
- хранилище блоков сохраняет две ветки без выбора основной;
- выбор ветки принимает решение по данным механизма согласования;
- Spring-обвязка переводит Spring-состояние в вид состояния для согласования;
- сеть доставляет сообщение без знания контрактов.

## Итог

Согласование, исполнение и состояние не должны быть полностью изолированы по
данным. Но они должны быть изолированы по ответственности.

Правильная архитектура признает связность и оформляет ее через явные порты.
Неправильная архитектура прячет связность в новый центральный управляющий
объект.
