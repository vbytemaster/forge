# Деление Savanna

## Назначение

Документ фиксирует, как Savanna должна стать готовым компонентом конструктора
блокчейнов и при этом не превратить FCL в Spring-специфичный блокчейн.

Ключевое решение:

```text
Общее ядро Savanna - будущий опциональный компонент FCL.
Spring-совместимая обвязка Savanna - компонент репы blockchain.
```

## Почему Savanna должна быть готовым компонентом

Если пользователь конструктора должен сам реализовывать Savanna с нуля, это не
конструктор. Это только набор низкоуровневых библиотек.

Конструктор должен давать готовые крупные детали:

- механизм голосования;
- проверку кворума;
- правила окончательности;
- проверку сообщений;
- обработку предложений;
- выбор основной ветки;
- состояние механизма согласования;
- тесты корректности.

Пользователь должен подключать Savanna и писать только обвязку своей цепи:

- откуда брать участников;
- какие данные блока подписываются;
- как политика участников хранится в состоянии;
- какие сетевые сообщения используются;
- как результаты окончательности отображаются в цепь.

## Почему нельзя положить Spring-детали в ядро Savanna

Savanna как алгоритм согласования должна быть пригодна для разных цепей. Если
ядро будет знать Spring-таблицы и Spring-заголовки, его нельзя будет
использовать вне Spring-совместимой цепи.

В ядре не должно быть:

- Spring-таблиц;
- Spring-действий;
- Spring-имен контрактов;
- Spring-специфичных расширений заголовка;
- Spring-специфичного внешнего программного интерфейса;
- правил переноса Spring-контрактов;
- продуктовых настроек будущей сети.

Эти вещи должны находиться в Spring-обвязке.

## Что живет в общем ядре Savanna

Общее ядро Savanna должно владеть только нейтральной логикой механизма
согласования.

### Голоса

Ядро должно уметь:

- принять голос;
- проверить форму голоса;
- проверить подпись через переданный криптографический интерфейс;
- определить вес голоса;
- определить, относится ли голос к известному предложению;
- отбрасывать дубли;
- отбрасывать конфликтующие голоса по правилам алгоритма.

Ядро не должно знать, из какой Spring-таблицы взят участник.

### Кворум

Ядро должно уметь:

- считать общий вес участников;
- считать собранный вес;
- определять, достигнут ли кворум;
- возвращать доказательство достижения кворума;
- не принимать голоса от неизвестных участников.

Ядро не должно знать, как именно цепь хранит список участников.

### Окончательность

Окончательность - это состояние, при котором блок больше нельзя откатить без
нарушения правил механизма согласования.

Ядро должно уметь:

- определить, что блок достиг окончательности;
- запретить конфликтующее окончательное решение;
- выдать обновление окончательности узлу;
- хранить минимальное внутреннее состояние, нужное для проверки будущих
  решений.

Ядро не должно само писать блок в Spring-журнал.

### Предложения блоков

Ядро должно уметь:

- проверить, что предложение относится к допустимому родителю;
- проверить данные согласования в предложении;
- определить, можно ли голосовать за предложение;
- вернуть причину отказа.

Ядро не должно исполнять транзакции предложения.

### Выбор основной ветки

Ядро должно уметь:

- сравнить кандидатов с точки зрения правил Savanna;
- учитывать окончательность;
- запрещать откат за окончательный блок;
- вернуть решение, какую ветку можно считать основной.

Ядро не должно хранить все блоки вместо хранилища блоков.

### Сетевые сообщения

Ядро может описывать типы сообщений согласования:

- предложение;
- голос;
- доказательство кворума;
- обновление окончательности;
- запрос недостающих данных.

Но привязка этих сообщений к конкретным сетевым темам и форматам цепи должна
быть снаружи.

## Что живет в `blockchain`

Репа `blockchain` держит Spring-совместимую обвязку Savanna.

Она отвечает за:

- чтение политики участников из Spring-совместимого состояния;
- перевод политики участников в нейтральный вид для ядра Savanna;
- построение Spring-совместимых данных заголовка;
- проверку Spring-совместимых расширений заголовка;
- упаковку и распаковку сетевых сообщений конкретной цепи;
- отображение окончательности в Spring-совместимое состояние;
- совместимость внешних интерфейсов;
- тесты переноса контрактов и блоков.

## Как выглядит связь ядра и обвязки

```text
Spring-совместимое состояние
  -> Spring-обвязка читает нужные таблицы
  -> Spring-обвязка строит нейтральный вид политики
  -> Ядро Savanna проверяет голоса и кворум
  -> Ядро Savanna возвращает решение
  -> Spring-обвязка применяет решение к цепи
```

Ядро не читает Spring-состояние напрямую. Оно получает уже подготовленный вид.

## Что должен уметь другой блокчейн

Другой блокчейн должен использовать Savanna без Spring.

Для этого он должен предоставить:

- свой вид участников;
- свой способ проверки ключей;
- свой формат блока;
- свой способ включить данные согласования в заголовок;
- свой способ хранить окончательность;
- свои сетевые сообщения или свою упаковку сообщений.

Если для этого приходится притворяться Spring-цепью, деление выполнено неверно.

## Интерфейсное правило

Общее ядро Savanna должно зависеть от абстрактных данных:

- идентификатор участника;
- вес участника;
- публичный ключ;
- идентификатор блока;
- номер блока;
- идентификатор родителя;
- голос;
- доказательство кворума;
- состояние окончательности.

Spring-обвязка должна зависеть от Spring-совместимых данных:

- Spring-совместимый заголовок;
- Spring-совместимое действие;
- Spring-совместимая таблица политики;
- Spring-совместимая упаковка;
- Spring-совместимый внешний ответ.

## Ошибки, которые нужно предотвратить

### Ошибка 1: ядро знает Spring-таблицу

Это сразу делает ядро непереносимым.

### Ошибка 2: обвязка заново реализует кворум

Тогда Savanna перестает быть готовым компонентом. Обвязка должна переводить
данные, а не дублировать алгоритм.

### Ошибка 3: выбор ветки живет в хранилище

Хранилище блоков не должно решать, какая ветка правильная. Для Savanna это
решение связано с окончательностью и голосами.

### Ошибка 4: внешний программный интерфейс напрямую дергает ядро Savanna

Внешний программный интерфейс должен ходить через прикладной слой узла. Иначе
наружу начнут протекать внутренние детали механизма согласования.

### Ошибка 5: Spring-обвязка становится новым управляющим центром

Обвязка должна связывать форматы, а не владеть всей цепью.

## Тестовые требования

Для общего ядра Savanna нужны тесты:

- голос неизвестного участника отклоняется;
- повторный голос не увеличивает вес;
- конфликтующий голос обрабатывается по правилам алгоритма;
- кворум достигается только при достаточном весе;
- окончательный блок нельзя заменить конфликтующим блоком;
- выбор ветки учитывает окончательность;
- ядро работает на искусственной цепи без Spring.

Для Spring-обвязки нужны тесты:

- Spring-политика участников переводится в нейтральный вид;
- Spring-совместимый заголовок получает правильные данные согласования;
- Spring-совместимый блок проверяется через ядро Savanna;
- окончательность отображается в состояние цепи;
- контрактные форматы не ломаются;
- сетевое сообщение цепи переводится в сообщение ядра и обратно.

## Критерий успеха

Деление Savanna считается правильным, если:

- ядро можно описать без слова Spring;
- Spring-обвязка не реализует алгоритм заново;
- другой блокчейн может использовать ядро Savanna;
- Spring-совместимая цепь получает готовый механизм согласования;
- FCL остается универсальным;
- `blockchain` сохраняет полный контроль над совместимостью.
