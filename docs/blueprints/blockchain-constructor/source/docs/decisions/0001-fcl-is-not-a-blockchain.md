# Решение 0001: FCL не является блокчейном

Дата: 2026-06-09

## Решение

FCL не превращаем в блокчейн и не делаем его Spring-специфичным проектом.

FCL остается универсальным фундаментом для сложного программного обеспечения:
сервисов, сетевых приложений, блокчейнов, DePIN-систем и других распределенных
продуктов.

Блокчейн-конструктор строится поверх FCL как опциональная вертикаль. Конкретная
Spring-совместимая цепь живет в репе `blockchain`.

## Контекст

Будущий блокчейн должен заменить Spring. Для этого нужно сохранить важную
совместимость:

- переносимость контрактов;
- совместимые форматы действий;
- совместимые транзакции;
- совместимые блоки там, где это требуется;
- совместимое поведение внешних инструментов.

Но Spring в текущем виде является монолитом. Если перенести его один к одному,
мы получим новый монолит. Если положить этот монолит в FCL, FCL станет
блокчейном и потеряет ценность как общий фундамент.

## Почему это важно

Нейтральность FCL нужна по пяти причинам.

### 1. FCL должен быть полезен не только блокчейнам

Пользователь FCL может строить обычный сетевой сервис. Он не должен видеть
Spring-таблицы и правила блоков в публичной модели.

### 2. Общие библиотеки должны развиваться независимо

Сеть, криптография, конфигурация и приложение не должны ломаться из-за изменения
конкретного блокчейна.

### 3. Совместимость нельзя путать с архитектурой

Spring задает, какие форматы и поведения нужно сохранить. Но это не значит, что
его внутреннее устройство нужно повторять.

### 4. Конструктор должен позволять разные цепи

Если FCL станет Spring-цепью, другой блокчейн не сможет удобно использовать его
как конструктор.

### 5. Savanna должна быть готовым компонентом, но не Spring-модулем

Savanna может стать повторно используемым механизмом согласования. Для этого ее
ядро должно быть нейтральным.

## Что разрешено в FCL

В FCL можно добавлять только то, что описывается без Spring- и продуктовых
понятий.

Разрешены:

- приложение;
- подключаемые модули;
- сеть;
- сообщения;
- протоколы;
- программные интерфейсы;
- конфигурация;
- криптография;
- сериализация;
- общие ошибки;
- диагностика;
- состояние;
- журнал изменений;
- голоса;
- кворум;
- окончательность;
- общий интерфейс механизма согласования;
- общее ядро Savanna.

## Что запрещено в FCL

В FCL запрещено добавлять как обязательную часть:

- Spring-таблицы;
- Spring-действия;
- Spring-специфичные заголовки блоков;
- Spring-совместимые внешние ответы;
- имена контрактов конкретной цепи;
- продуктовые правила ресурсов;
- правила переноса конкретных контрактов;
- Storlane-термины;
- конкретную сборку будущего блокчейна;
- сетевые сообщения, имеющие смысл только для Spring-совместимой цепи.

## Последствия

### Положительные

- FCL остается универсальным.
- `blockchain` получает свободу строить Spring-совместимость без загрязнения
  FCL.
- Savanna можно сделать готовым компонентом конструктора.
- Другие продукты смогут использовать FCL без блокчейн-багажа.
- Будущая нарезка работ станет понятнее.

### Отрицательные

- Появляется дополнительная граница между FCL и `blockchain`.
- Нужно писать обвязки вместо прямого доступа к Spring-структурам.
- Некоторые решения нельзя будет быстро "просто положить в FCL".

Эти издержки принимаются. Они дешевле, чем превращение FCL в новый монолит.

## Практическое правило

Если компонент нельзя объяснить без слов Spring, Storlane или имени конкретного
контракта, он не должен попадать в FCL.

Если компонент можно использовать в неблокчейн-распределенной системе, он может
быть кандидатом в FCL.
