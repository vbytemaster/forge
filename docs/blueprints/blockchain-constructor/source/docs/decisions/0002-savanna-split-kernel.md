# Решение 0002: Savanna делится на ядро и Spring-обвязку

Дата: 2026-06-09

## Решение

Savanna должна быть готовым повторно используемым механизмом согласования, но
без Spring-специфичных деталей в общем ядре.

Деление:

- общее ядро Savanna - будущий опциональный компонент FORGE;
- Spring-совместимая обвязка Savanna - компонент репы `blockchain`.

## Контекст

Нам нужен конструктор блокчейнов. Если каждый пользователь должен заново
реализовывать Savanna, это не конструктор, а только низкоуровневый набор
библиотек.

С другой стороны, если Savanna сразу положить в FORGE вместе со Spring-таблицами,
Spring-заголовками и Spring-правилами, FORGE станет блокчейн- и
Spring-специфичным.

Поэтому Savanna нужно разделить.

## Почему Savanna не пишется каждым пользователем заново

Механизм согласования - сложная и критичная часть блокчейна. Ошибка в нем может
сломать безопасность всей сети.

Конструктор должен давать готовую реализацию:

- проверки голосов;
- подсчета кворума;
- правил окончательности;
- защиты от конфликтующих решений;
- обработки предложений блоков;
- выбора основной ветки;
- тестов корректности.

Пользователь конструктора должен описывать свою цепь, а не переписывать алгоритм
согласования.

## Почему Spring-совместимость не должна жить в общем ядре

Spring-совместимость - это обвязка вокруг алгоритма. Она отвечает за то, как
данные Spring-цепи переводятся в данные механизма согласования.

Если эта обвязка попадет в ядро, ядро нельзя будет использовать в другой цепи.

В ядре не должно быть:

- Spring-таблиц;
- Spring-имен;
- Spring-специфичных расширений заголовка;
- Spring-совместимых внешних ответов;
- правил переноса контрактов;
- продуктовых правил будущей сети.

## Что делает ядро

Ядро Savanna отвечает за нейтральный алгоритм:

- участники;
- веса участников;
- голоса;
- подписи;
- кворум;
- предложения блоков;
- доказательства кворума;
- окончательность;
- выбор основной ветки;
- ошибки согласования.

Ядро не знает, как именно цепь хранит участников.

## Что делает Spring-обвязка

Spring-обвязка отвечает за перевод между Spring-совместимой цепью и ядром:

- читает Spring-совместимое состояние;
- извлекает политику участников;
- переводит участников в нейтральный вид;
- строит данные заголовка;
- проверяет Spring-совместимые поля;
- применяет решение окончательности к цепи;
- поддерживает совместимые ответы программного интерфейса.

Обвязка не должна заново реализовывать алгоритм Savanna.

## Как это сохраняет FORGE универсальным

FORGE получает готовый механизм согласования, но не получает Spring как
обязательную модель.

Другие пользователи смогут взять ядро Savanna и написать свою обвязку:

- для другой модели блоков;
- для другой системы контрактов;
- для другой политики участников;
- для другой упаковки сообщений;
- для другой сети.

При этом Spring-совместимая цепь получает не пустой интерфейс, а настоящий
готовый механизм.

## Последствия

### Положительные

- Savanna становится частью конструктора.
- FORGE не превращается в Spring.
- Spring-совместимость остается контролируемой.
- Алгоритм можно тестировать отдельно от Spring.
- Другие цепи могут использовать Savanna.

### Отрицательные

- Нужно проектировать явные порты между ядром и обвязкой.
- Нужно писать тесты и для ядра, и для обвязки.
- Нельзя быстро зашить Spring-таблицу прямо в механизм согласования.

Эти издержки принимаются ради чистой архитектуры и повторного использования.

## Критерий выполнения решения

Решение выполнено, если:

- ядро Savanna можно описать без Spring;
- Spring-обвязка не содержит собственной реализации кворума;
- тесты ядра не требуют Spring-цепи;
- тесты обвязки доказывают Spring-совместимость;
- FORGE остается пригодным для неблокчейн-проектов.
