# Словарь

Документ закрепляет русские термины для архитектуры конструктора блокчейнов.
Английские термины оставлены только там, где это имя проекта, протокола,
общеупотребимое сокращение или будущий поисковый якорь.

## FCL

FCL - Foundation Core Libraries, фундаментальные C++ библиотеки.

В этих документах FCL означает нейтральный набор библиотек и прикладных слоев
для сложного программного обеспечения. FCL не означает блокчейн.

## Конструктор блокчейнов

Набор общих компонентов, из которых можно собрать конкретную цепь.

Конструктор должен давать готовые крупные детали, но не должен навязывать
Spring-совместимость всем пользователям FCL.

## Цепь

Последовательность блоков, связанных ссылками на предыдущий блок.

В документах слово "цепь" используется вместо английского слова blockchain там,
где не нужно называть репу или общий класс технологии.

## Блок

Пакет транзакций и служебных данных, который продолжает цепь.

Блок обычно содержит:

- номер;
- ссылку на родителя;
- заголовок;
- транзакции;
- подпись или подписи;
- данные механизма согласования.

## Транзакция

Запрос на изменение состояния цепи.

Транзакция может содержать одно или несколько действий, подписи, срок действия и
данные, нужные для проверки.

## Действие

Одна прикладная операция внутри транзакции.

Для Spring-совместимости важно сохранить формат действий и порядок полей там,
где это видят контракты.

## Состояние цепи

Данные, которые получаются после последовательного применения блоков.

Состояние включает контрактные таблицы, системные данные, политики участников и
другие сведения, нужные для проверки и исполнения.

## Исполнение

Применение транзакций к состоянию.

Исполнение проверяет транзакции, запускает контрактный код, меняет состояние и
создает результат блока.

## Результат исполнения

Данные, которые появляются после применения блока:

- новое состояние;
- квитанции;
- корни данных;
- ошибки;
- признаки изменения политики участников;
- данные для заголовка.

## Квитанция

Запись о результате исполнения транзакции или действия.

Квитанция нужна для проверки, внешних инструментов и совместимости поведения.

## Механизм согласования

Часть узла, которая помогает участникам сети прийти к одному решению о блоках.

Механизм согласования проверяет предложения, голоса, кворум, окончательность и
правила выбора ветки.

## Предложение блока

Блок или заготовка блока, которую участник предлагает другим участникам сети.

Предложение должно пройти проверку исполнения и проверку механизма
согласования.

## Голос

Подписанное сообщение участника о поддержке определенного блока или решения.

Голос учитывается механизмом согласования.

## Кворум

Достаточное количество или достаточный вес голосов для принятия решения.

Точный порог зависит от механизма согласования.

## Окончательность

Состояние, при котором блок считается необратимым по правилам механизма
согласования.

После окончательности нельзя перейти на конфликтующую ветку без нарушения
правил.

## Развилка

Ситуация, когда у одного блока появляется несколько разных продолжений.

Развилка не всегда ошибка. Узел должен уметь хранить и проверять несколько
веток, пока механизм согласования и выбор ветки не определят основную.

## Выбор основной ветки

Правило, по которому узел определяет, какая из допустимых веток считается
основной.

Выбор основной ветки связан с механизмом согласования и не должен жить внутри
обычного хранилища блоков.

## Хранилище блоков

Компонент, который записывает и читает блоки.

Хранилище блоков не должно самостоятельно решать, какая ветка основная.

## Пул транзакций

Компонент, который хранит транзакции до включения в блок.

Пул отвечает за прием, удаление дублей, срок жизни, порядок выбора и удаление
транзакций после принятия блока.

## Производитель блока

Участник или компонент узла, который собирает новый блок.

Производитель берет транзакции из пула, запускает исполнение, добавляет данные
механизма согласования и подписывает блок.

## Переносимость контрактов

Способность контракта, написанного для Spring, работать в новой цепи без
переписывания или с минимально контролируемыми изменениями.

Для этого нужны совместимые форматы данных, правила исполнения и видимые
контракту системные значения.

## Двоичный интерфейс контрактов

Правила, по которым контракт видит входные и выходные данные на уровне байтов.

В английских документах часто используется сокращение ABI. В этих документах
используем русское название, а сокращение указываем только при необходимости.

## WASM

Формат байткода WebAssembly, в котором могут выполняться контракты.

Сокращение WASM оставляем как общеупотребимое имя технологии.

## Одноранговая сеть

Сеть, где узлы обмениваются сообщениями напрямую друг с другом.

В английских текстах часто используется P2P. В этих документах используем
"одноранговая сеть", а P2P оставляем только как имя существующего слоя FCL.

## API

Программный интерфейс для внешних клиентов или других компонентов.

Сокращение API общеупотребимо, но при первом употреблении в документах желательно
писать "программный интерфейс".

## Обвязка

Слой, который переводит данные одной модели в данные другой модели.

Например, Spring-обвязка Savanna переводит Spring-совместимое состояние
участников в нейтральный вид, понятный ядру Savanna.

## Ядро

Часть компонента, где находится его общая логика без продуктовых деталей.

Например, ядро Savanna должно знать голоса, кворум и окончательность, но не
Spring-таблицы.

## Донор

Существующий проект, документация или реализация, из которых мы берем
архитектурный урок.

Донор не обязательно становится зависимостью.

## Spring

Текущая система, которую будущий блокчейн должен заменить.

В этих документах Spring - цель совместимости, но не образец внутренней
архитектуры.

## Savanna

Механизм согласования, который должен стать готовым повторно используемым
компонентом конструктора.

Savanna делится на общее ядро и Spring-совместимую обвязку.

## Cosmos SDK

Проект-донор модульной сборки блокчейнов.

Мы берем из него идею модулей и явного владения состоянием, но не копируем
техническую реализацию.

## CometBFT ABCI

Проект-донор границы между приложением и механизмом согласования.

ABCI означает прикладной интерфейс блокчейна. Мы берем идею явной границы, но не
копируем конкретный набор методов.

## Ethereum Engine API

Проект-донор границы между исполнением и механизмом согласования.

Мы берем идею жесткого разделения ролей и явных статусов проверки, но не берем
форматы Ethereum.
