# План будущей нарезки FCL

## Назначение

Документ описывает, как позже нарезать работы в FCL. Сейчас FCL репа не
меняется. Этот план нужен, чтобы будущие задачи не начинались с повторного
обсуждения границ.

Главное правило:

```text
В FCL переносим только общие строительные блоки.
В blockchain оставляем конкретную Spring-совместимую цепь.
```

## Предварительная фаза: закрепить архитектуру в `blockchain`

Перед началом разработки в FCL нужно завершить документационную базу:

- граница FCL;
- устройство конструктора;
- разбор Spring;
- граница согласования;
- деление Savanna;
- потоки данных;
- донорская карта;
- словарь терминов.

Готовность фазы:

- документы согласованы;
- устаревшие черновики удалены;
- есть явное разделение FCL / `blockchain` / Spring-совместимость;
- есть список будущих блоков FCL;
- нет требования менять FCL немедленно.

## Блок 1: Общие понятия цепи без Spring

### Цель

Описать минимальные нейтральные понятия, которые нужны механизму согласования и
сборке цепи, но не содержат Spring.

### Возможное место в FCL

Будущий опциональный слой общих распределенных примитивов. Окончательное имя
нужно выбрать при разработке, не раньше.

### Содержание

- идентификатор блока;
- номер блока;
- ссылка на родителя;
- статус проверки;
- статус наличия данных;
- статус окончательности;
- причина отказа;
- результат выбора ветки.

### Что не переносить в FCL

- Spring-заголовок;
- Spring-транзакцию;
- Spring-действие;
- Spring-таблицу;
- контрактные типы.

### Тесты

- типы можно использовать без `blockchain`;
- нет зависимости от Spring-форматов;
- ошибки имеют явные категории;
- данные можно использовать в искусственной цепи.

## Блок 2: Интерфейс механизма согласования

### Цель

Описать общий порт, через который цепь взаимодействует с механизмом
согласования.

### Содержание

- проверка предложения блока;
- проверка голоса;
- построение данных согласования для заголовка;
- обновление окончательности;
- выбор основной ветки;
- перечень сетевых сообщений механизма согласования.

### Что не переносить в FCL

- чтение Spring-таблиц;
- Spring-совместимую упаковку сообщений;
- Spring-совместимый внешний программный интерфейс;
- правила контрактов.

### Тесты

- искусственный механизм согласования можно подключить к тестовой цепи;
- механизм возвращает явные решения;
- статусы "принято", "отклонено", "не хватает данных" различимы;
- интерфейс не требует Spring.

## Блок 3: Виды состояния для согласования

### Цель

Дать механизму согласования только нужное чтение состояния, без доступа к
полной базе цепи.

### Содержание

- текущая политика участников;
- будущая политика участников;
- последний окончательный блок;
- текущая голова;
- параметры кворума;
- ключи участников;
- сведения о поколении политики.

### Что не переносить в FCL

- конкретную Spring-схему хранения;
- контрактные таблицы;
- правила переноса контрактов;
- прямую запись состояния.

### Тесты

- вид состояния доступен только для чтения;
- механизм согласования не может менять состояние;
- Spring-обвязка может построить такой вид из Spring-совместимого состояния;
- искусственная цепь может построить такой вид без Spring.

## Блок 4: Результат исполнения для согласования

### Цель

Описать, какие данные исполнитель передает механизму согласования после
применения блока.

### Содержание

- корень транзакций;
- корень действий;
- сводка квитанций;
- признаки изменения политики;
- данные заголовка;
- ошибка исполнения;
- статус проверки полезной нагрузки блока.

### Что не переносить в FCL

- Spring-формат квитанции;
- Spring-порядок полей;
- контрактную машину;
- конкретную базу состояния.

### Тесты

- механизм согласования может проверить блок по результату исполнения;
- исполнитель не вызывает сетевое голосование;
- результат можно построить в тестовой цепи без Spring.

## Блок 5: Общее ядро Savanna

### Цель

Реализовать Savanna как готовый повторно используемый механизм согласования.

### Содержание

- участники;
- веса;
- голоса;
- проверка подписей;
- кворум;
- предложения блоков;
- доказательство кворума;
- окончательность;
- выбор основной ветки;
- ошибки согласования;
- внутреннее состояние механизма.

### Что не переносить в FCL

- Spring-обвязку;
- Spring-заголовки;
- Spring-таблицы политики;
- Spring-совместимый программный интерфейс;
- перенос контрактов.

### Тесты

- кворум считается корректно;
- повторный голос не увеличивает вес;
- неизвестный участник отклоняется;
- конфликтующие голоса обрабатываются по правилам;
- окончательный блок нельзя заменить конфликтующим;
- ядро работает на искусственной цепи без Spring.

## Блок 6: Каркас сетевых сообщений согласования

### Цель

Описать общую форму сообщений механизма согласования без привязки к конкретной
цепи.

### Содержание

- предложение;
- голос;
- доказательство кворума;
- запрос недостающих данных;
- ответ с недостающими данными;
- базовая проверка размера и формы.

### Что не переносить в FCL

- Spring-совместимую упаковку сообщений;
- конкретные сетевые темы будущей цепи;
- внешний программный интерфейс узла.

### Тесты

- сообщения можно передать через существующий слой одноранговой сети FCL;
- сообщения не знают Spring-типов;
- неправильный размер и форма отклоняются до дорогой проверки.

## Блок 7: Spring-обвязка в `blockchain`

### Цель

Связать Spring-совместимую цепь с общим ядром Savanna.

### Место

Только `blockchain`, не FCL.

### Содержание

- чтение Spring-совместимой политики участников;
- перевод участников в нейтральный вид;
- построение данных заголовка;
- проверка Spring-совместимого блока;
- применение окончательности;
- упаковка сетевых сообщений цепи;
- совместимые ответы программного интерфейса.

### Тесты

- Spring-политика правильно переводится в вид ядра;
- блок с корректными данными проходит проверку;
- блок с неверными данными отклоняется;
- контрактная совместимость не ломается;
- окончательность отображается в цепь.

## Блок 8: Пул транзакций как каркас

### Цель

Выделить общий каркас пула транзакций, если он окажется достаточно нейтральным.

### Содержание

- прием;
- удаление дублей;
- срок жизни;
- приоритет;
- вытеснение;
- выдача кандидатов;
- реакция на принятие блока;
- реакция на окончательность.

### Что оставить в `blockchain`

- Spring-проверку транзакции;
- Spring-формат транзакции;
- правила ресурсов;
- совместимость внешних ошибок.

### Тесты

- дубли не принимаются;
- истекшие транзакции удаляются;
- принятые транзакции удаляются после блока;
- окончательные транзакции удаляются без возврата;
- каркас работает на искусственном типе транзакции.

## Блок 9: Хранилище блоков как порт

### Цель

Описать общий порт хранения блоков без правил выбора ветки.

### Содержание

- запись блока;
- чтение по номеру;
- чтение по идентификатору;
- наличие блока;
- диапазон блоков;
- отметка необратимости;
- данные для повторного проигрывания.

### Что оставить в `blockchain`

- Spring-журнал блоков;
- Spring-формат записи;
- совместимость старых файлов;
- продуктовые правила восстановления.

### Тесты

- можно хранить две ветки;
- хранилище не выбирает основную ветку;
- отметка необратимости не равна выбору ветки;
- данные читаются после перезапуска.

## Блоки, которые не должны попадать в FCL

Следующие вещи остаются в `blockchain`:

- Spring-совместимая модель контрактов;
- Spring-совместимые действия;
- Spring-совместимые таблицы;
- Spring-совместимые внешние программные интерфейсы;
- перенос конкретных контрактов;
- продуктовая конфигурация будущей сети;
- конкретная сборка узла;
- конкретная экономика или правила ресурсов.

## Порядок работ

Рекомендуемый порядок:

1. Согласовать документы в `blockchain`.
2. Создать ветку FCL только после отдельной команды.
3. Начать с нейтральных типов и интерфейса механизма согласования.
4. Добавить виды состояния и результата исполнения.
5. Реализовать искусственный тестовый механизм согласования.
6. Реализовать ядро Savanna.
7. Подключить Spring-обвязку в `blockchain`.
8. Расширить сетевые сообщения.
9. Выделять пул и хранилище только после проверки границ.

## Критерии готовности к началу FCL-разработки

Можно начинать FCL-разработку, когда:

- ясно, какие типы нейтральны;
- ясно, какие типы Spring-специфичны;
- есть минимальный интерфейс механизма согласования;
- есть тестовый сценарий без Spring;
- понятно, как Spring-обвязка будет подключаться снаружи;
- нет необходимости менять слой одноранговой сети FCL для первого этапа;
- выбран первый маленький блок, который можно проверить отдельно.

## Критерии отказа от переноса в FCL

Компонент не переносим в FCL, если:

- его нельзя назвать без Spring;
- его нельзя протестировать без Spring;
- он нужен только для переносимости конкретных контрактов;
- он содержит продуктовые правила;
- он заставляет неблокчейн-пользователей FCL видеть блокчейн-модель.
